# NRIN — MATCH PRESENTATION V1

## Purpose

The match interface is not a list of results.

It is:
**a guided introduction to a place that can help**

The system must:
- reduce fear
- increase trust
- remove cognitive overload
- feel calm, certain, and human

The system must NOT:
- feel algorithmic
- expose scoring mechanics
- resemble dashboards or AI outputs

---

## Core Principle

### The facility presents itself  
Not the system presenting the facility

Shift from:
- “we matched this for you”

To:
- “this is a place that fits your situation”

---

## Information Hierarchy

### Level 1 — Identity
- Facility name
- Visual identity (image or logo)
- Location (optional, subtle)

### Level 2 — Human framing
- One calm, reassuring statement
- Interprets fit in human terms

Examples:
- “This facility aligns closely with the level of care you need.”
- “A structured environment with support for both detox and ongoing care.”

Do not use:
- “Detected signals include…”
- “Programs: 100”
- “Confidence 10”

### Level 3 — Action
Primary CTA should feel:
- safe
- reversible
- non-committal

Preferred labels:
- “Continue”
- “See if this is right for you”
- “Explore this option”

### Level 4 — Explanation
Secondary CTA:
- “Why this may be a strong fit for you”

Hidden by default.
Expanded only on user intent.

Tone:
- calm
- specific
- compassionate

### Level 5 — Navigation
- Previous / Next
- minimal visual weight
- no dashboard feel

---

## Visual Model

### Surface
Not a “card” in the SaaS sense.

Use:
- soft container
- minimal borders
- minimal nested boxes
- restrained controls

Avoid:
- badge clusters
- stacked chips
- debug-style sections
- floating widget feel

### Spacing
- generous vertical rhythm
- breathing room between sections
- avoid cramped signal piles

### Typography
- strong, confident title
- subdued support text
- limited on-screen information
- no dense metadata

### Visual Anchor
Each facility should have one anchor, in priority order:
1. real image
2. brand image
3. logo
4. initials

Rule:
**logos are not hero content**

---

## Explicitly Forbidden

Do not show:
- score math
- badges like “Programs 100”
- “Insurance 15”
- “Specialties 14”
- “Confidence 10”
- system language like:
  - detected
  - signals
  - confidence
  - raw insurer snake_case

Do not create:
- admin dashboard feeling
- AI copilot feeling
- audit-log feeling

---

## Emotional Tone

The user may be:
- overwhelmed
- afraid
- uncertain

The interface must feel:
- grounded
- steady
- non-urgent
- quietly confident

Never:
- pushy
- overly analytical
- machine-explanatory

---

## Interaction Model

### Default state
Show only:
- identity
- calm framing
- action

### On intent
Reveal:
- explanation
- evidence
- source links

Hierarchy:
persuasion → explanation → verification

---

## Data Philosophy

The system may compute:
- scores
- signals
- ranking

But the UI should show only:
**what a trusted human would naturally say**

---

## Near-Term Build Direction

### V1
- facility-first presentation
- explanation on demand
- no score/badge leakage
- logo supported
- hero image optional

### V2
- real asset layer
- brochure imagery
- richer narrative explanation
- insurer display polish
- intake-to-match personalization

---

## Design Goal

The user should feel:

“This looks like a real place I could go.”

Not:

“This is a system recommending something.”

---

## Strategic Reminder

NRIN is not a matching UI.

It is:
**the front door to a recovery decision**
